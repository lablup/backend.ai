Replace the misleading `desired_replica_count` / `desired_replicas` input field with `replica_count` on deployment create/update across REST v1, GraphQL v2, and CLI. The previous name accepted an autoscaling-internal target that was overwritten on the next reconcile tick; clients now set the user-controllable `replica_count` directly, matching the data layer and the `endpoints.replicas` column. Response DTOs continue to expose `desired_replica_count` as a read-only view of the current scheduling state.
